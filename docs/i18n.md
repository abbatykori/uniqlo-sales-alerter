# i18n workflow

The app's user-facing strings live in `gettext` catalogues under
`src/uniqlo_sales_alerter/i18n/locale/<lang>/LC_MESSAGES/messages.po`. The
foundation phase ships one language (`en`); additional languages are a
translation-only PR once UI strings are wrapped in `_(...)`.

## Adding or updating strings

After adding new wrapped strings (`_("...")` in Python or `{{ _("...") }}` in
Jinja templates) anywhere under `src/`, regenerate the catalogues:

```bash
# 1. Extract source strings into the template (.pot)
pybabel extract -F babel.cfg -o src/uniqlo_sales_alerter/i18n/messages.pot src/

# 2. Merge new entries into each language catalogue
pybabel update \
    -i src/uniqlo_sales_alerter/i18n/messages.pot \
    -d src/uniqlo_sales_alerter/i18n/locale \
    -l en

# 3. Compile to the binary format the runtime reads
pybabel compile -d src/uniqlo_sales_alerter/i18n/locale
```

The compiled `.mo` files are git-ignored. The Docker build runs `pybabel
compile` in the builder stage, so containers always ship freshly compiled
catalogues.

## Adding a new language

Drop the language code into the locale directory and run `pybabel init`:

```bash
pybabel init \
    -i src/uniqlo_sales_alerter/i18n/messages.pot \
    -d src/uniqlo_sales_alerter/i18n/locale \
    -l <code>
```

Translate the resulting `messages.po`, then run `pybabel compile` as above.
No code change is required: `get_translator(<code>)` will pick the new
catalogue up automatically. Set `ui_language: "<code>"` in `config.yaml` (or
`UI_LANGUAGE=<code>`) to use it.
